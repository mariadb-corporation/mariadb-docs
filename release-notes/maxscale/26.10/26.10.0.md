# MariaDB MaxScale 26.10.0 Release Notes

Release 26.10.0 is a GA release. It was released on X-Y-Z.

This document describes the changes in release 26.10, when compared to
release 25.10.

For any problems you encounter, please consider submitting a bug
report on [our Jira](https://jira.mariadb.org/projects/MXS).

## Changed Features

### [MXS-6550](https://jira.mariadb.org/browse/MXS-6550) ISO 8601 timestamps in MaxScale log messages

Log messages now use a ISO 8601 timestamp which is a standardized, easy to parse
time format that includes the UTC timezone offset for easier cross-region log
analysis. To use the old legacy timestamp format, add
`log_timestamp_format=datetime` under the `[maxscale]` section. By default, a
warning is logged on startup for existing installations. To silence it, add
`log_timestamp_format=iso_8601` under the `[maxscale]` section.

## New Features

### [MXS-6102](https://jira.mariadb.org/browse/MXS-6102) If a server keeps sending routing replies, label it as Running regardless of monitoring result

A monitored server may remain in running status if the connection failure was
due to connection timeout and if MaxScale has recently received data from the
server in response to a client query. These conditions suggest that the server
is still running, but too busy to process MaxScale monitor connections.

### [MXS-5976](https://jira.mariadb.org/browse/MXS-5976) Add SNI support

Listeners now support [SNI](https://en.wikipedia.org/wiki/Server_Name_Indication).
SNI allows a client to connect to different MaxScale services through one
listener. This feature is configured with an external JSON file. See
[listener documentation](https://app.gitbook.com/s/0pSbu5DcMSW4KwAkUcmX/maxscale-management/deployment/maxscale-configuration-guide.md#sni_mapping_file)
for more information.

### [MXS-5594](https://jira.mariadb.org/browse/MXS-5976) Allow encrypted and unencrypted connections on the same listener

A listeners now accepts encrypted and unencrypted connections if configured with `ssl=allow`.
See [configuration quide](https://app.gitbook.com/s/0pSbu5DcMSW4KwAkUcmX/maxscale-management/deployment/maxscale-configuration-guide.md#tlsssl-encryption)
for more information.

### [MXS-6383](https://jira.mariadb.org/browse/MXS-6383) Support "allow" SSL-mode in server configuration

If a server is configured with `ssl=allow`, MaxScale uses TLS if server
supports it, but falls back to unencrypted connections if not.
See [configuration quide](https://app.gitbook.com/s/0pSbu5DcMSW4KwAkUcmX/maxscale-management/deployment/maxscale-configuration-guide.md#tlsssl-encryption)
for more information.

`ssl=allow` is the new default value, which means MaxScale defaults to
encrypted backend connections if available. This can increase CPU usage
slightly. To disable encryption for backend connections, set `ssl=disable`
for each server.

### [MXS-6233](https://jira.mariadb.org/browse/MXS-6233) External replication management in MariaDB Monitor

MariaDB Monitor supports external replication management by relying on the
results of another monitor. This allows the monitor to update the external
replication connection if the external primary goes down or its role changes.
See
[MariaDB Monitor documentation](https://app.gitbook.com/s/0pSbu5DcMSW4KwAkUcmX/reference/maxscale-monitors/mariadb-monitor.md#external-replication-management)
for more information.
